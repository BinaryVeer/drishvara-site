# Content Stage C12 — Content Asset Verification & Review Workflow Preview

Status: Content governance / read-only review workflow preview  
Phase: C-Content Governance  
Depends on: C01 through C11, I00  
Runtime impact: None  
Subscriber impact: None  
Public API impact: None  
ML impact: None  

## 1. Purpose

C12 creates a read-only verification and review workflow preview for the candidate content assets extracted in C11.

C12 does not approve assets, publish assets, mutate article files, mutate homepage featured reads, mutate sitemap, write final registries, activate admin review, activate Supabase, activate Auth, activate payment, fetch external APIs, crawl links, download images, create embeddings, train models, or write vector database records.

C12 converts the C11 candidate asset inventory into review workflow recommendations only.

## 2. Source File

C12 reads:

- data/content/content-asset-inventory-preview.json

This file was generated by C11 and is still preview-only.

## 3. Output File

C12 may write only this preview output:

- data/content/content-asset-verification-review-preview.json

This is not a live review queue.

This is not an approval registry.

This is not an ML eligibility report.

## 4. Relationship with C10 and C11

C10 defines the asset governance and ML eligibility rules.

C11 extracts candidate assets from existing static content data.

C12 prepares a verification and review workflow preview for those candidate assets.

C12 must preserve these defaults:

- ML-training eligibility is false;
- embedding eligibility is false;
- registry write is false;
- public approval is not granted;
- admin review remains disabled;
- Supabase remains disabled.

## 5. Review Workflow Doctrine

Every candidate asset should receive a preliminary review workflow recommendation.

A review workflow recommendation may include:

- source review;
- rights review;
- quality review;
- duplicate review;
- image review;
- Hindi readiness review;
- public-safety review;
- ML eligibility review;
- embedding eligibility review;
- human editorial review.

These are recommendations only.

## 6. Review Status Doctrine

C12 may assign preview review statuses:

- review_not_started;
- source_review_required;
- rights_review_required;
- quality_review_required;
- duplicate_review_required;
- image_review_required;
- hindi_review_required;
- public_safety_review_required;
- ml_eligibility_blocked;
- embedding_eligibility_blocked;
- ready_for_human_editorial_review;
- blocked_pending_basic_metadata.

No status in C12 means approved.

## 7. Review Priority Doctrine

C12 may assign preview priority:

- low;
- medium;
- high;
- blocked.

Priority is based on visible metadata hints only.

A candidate with missing path, missing title, or severe metadata gaps should be blocked pending basic metadata.

## 8. Basic Metadata Check Doctrine

C12 checks whether candidate assets appear to have:

- title;
- article path;
- category;
- summary;
- image;
- homepage featured observation;
- SEO reference observation.

These checks are preliminary and do not replace editorial review.

## 9. Eligibility Recommendation Doctrine

C12 must keep:

- ml_training_eligible: false
- embedding_eligible: false
- public_approval_granted: false
- registry_write_allowed: false

C12 may recommend future review paths but cannot mark assets as ML-ready or embedding-ready.

## 10. Human Review Doctrine

C12 must require human review before any of the following future actions:

- approve public-safe status;
- approve source and rights;
- approve image usage;
- approve ML-training eligibility;
- approve embedding eligibility;
- select canonical duplicate;
- publish or republish an asset;
- activate any admin workflow.

## 11. Review Action Options

Future reviewer actions may include:

- request_source_details;
- request_rights_details;
- request_image_review;
- request_quality_revision;
- request_duplicate_check;
- request_hindi_review;
- mark_reference_only;
- mark_internal_only;
- mark_candidate_for_public_review;
- keep_blocked;
- escalate_to_editorial_owner.

C12 does not perform these actions.

## 12. Explicit Exclusions

C12 does not:

- mutate article files;
- mutate data/article-index.json;
- mutate data/homepage-ui.json;
- mutate SEO metadata;
- mutate sitemap;
- write final content asset registry;
- write image asset registry;
- write verified link registry;
- write review queue records;
- activate admin review;
- activate manual override;
- activate selection memory;
- activate Supabase;
- activate Auth;
- activate payment;
- fetch external APIs;
- crawl links;
- download images;
- approve rights;
- approve sources;
- approve public-safe status;
- approve ML training;
- approve embedding;
- create embeddings;
- train models;
- write vector database records;
- activate public output;
- activate subscriber output.

## 13. Safety Doctrine

C12 must not imply that any article, image, or link is verified, approved, public-safe, ML-eligible, or embedding-eligible.

C12 must label all outputs as preview-only.

## 14. C12 Acceptance Criteria

C12 is complete when:

1. C12 document exists.
2. C12 registry exists.
3. C12 generator exists.
4. C12 validator exists.
5. C12 preview output exists.
6. Preview output is marked preview-only.
7. Review workflow recommendations are generated for candidate assets.
8. No asset is approved by C12.
9. ML-training eligibility remains false.
10. Embedding eligibility remains false.
11. Registry write remains false.
12. Public approval remains false.
13. Admin/Supabase/Auth/payment remain disabled.
14. validate:c12 passes.
15. validate:content passes.
16. validate:project passes.

## 15. C12 Status

C12 establishes Content Asset Verification & Review Workflow Preview.

C12 does not approve, publish, mutate, embed, train, or activate runtime.
