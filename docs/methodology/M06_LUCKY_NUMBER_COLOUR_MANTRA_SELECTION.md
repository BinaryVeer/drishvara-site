# M06 — Lucky Number / Colour / Mantra Selection Methodology

Status: Methodology/Governance only  
Phase: M-Methodology  
Depends on: M00, M01, M02, M03, M04, M04A, M05  
Runtime impact: None  
Subscriber impact: None  
Public guidance impact: None  

## 1. Purpose

M06 defines Drishvara’s methodology for future lucky number, lucky colour, and mantra selection.

M06 does not generate live lucky numbers, lucky colours, mantras, subscriber guidance, dashboard cards, premium outputs, Auth, Supabase, payment, external API fetch, or public dynamic guidance.

M06 is a symbolic guidance methodology layer only.

## 2. Core Doctrine

Lucky number, lucky colour, and mantra must be treated as symbolic, devotional, reflective, or curated guidance.

They must not be presented as deterministic fate calculation, guaranteed remedy, medical advice, legal advice, financial advice, or fear-based religious instruction.

M06 inherits M00 Sanskrit integrity and no-invented-mantra doctrine.

## 3. Explicit Exclusions

M06 does not implement lucky number runtime, lucky colour runtime, mantra runtime, mantra generation, subscriber dashboard card, premium guidance, personalized output, Auth, Supabase, payment, subscription entitlement, external API fetch, DOB prediction, Kundli generation, horoscope generation, automated astrological remedy, fear-based warning, or public guidance.

M06 does not create new Sanskrit text or invented mantras.

## 4. Relationship with Earlier Modules

M06 inherits M00 source doctrine, M00 Sanskrit integrity framework, M00 no-invented-mantra rule, and M00 privacy doctrine.

M06 inherits M05 consent-first, data minimization, uncertainty, profile completeness, and privacy methodology.

M06 may later consume M01 to M04 data only through reviewed, consented, and non-deterministic guidance layers.

M06 inherits M04A learning register and database tuning doctrine for validation and correction of symbolic guidance records.

## 5. Symbolic Guidance Doctrine

Lucky number and lucky colour must be labelled as symbolic guidance.

They may be based on reviewed rule families such as:

- generic daily curation;
- date-based symbolic rotation;
- weekday-based symbolic rotation;
- tithi-linked symbolic rotation;
- festival-linked symbolic rotation;
- profile-consented preference;
- manually curated editorial theme;
- reviewed traditional association.

No symbolic guidance may claim guaranteed success, guaranteed protection, guaranteed remedy, or exact prediction.

## 6. Lucky Number Methodology

Future lucky number selection must preserve:

- selection basis;
- input signals used;
- consent status;
- rule family;
- fallback rule;
- confidence label;
- explanation label;
- review status.

Allowed future lucky number bases:

- daily symbolic rotation;
- weekday symbolic rotation;
- tithi symbolic rotation;
- festival symbolic rotation;
- user preference;
- editorial curation;
- reviewed numerological association if source-reviewed.

Not allowed:

- claiming guaranteed fortune;
- claiming unavoidable destiny;
- using DOB without consent;
- using name to infer caste, religion, ethnicity, or identity;
- hidden premium manipulation;
- fear-based number warnings.

Allowed future lucky colour bases:

## 8. Mantra Selection Methodology

Mantra selection has stricter rules than number or colour.

A mantra may be selected only if the mantra exists in an approved source record.

Every future mantra record must include:

- mantra_id;
- deity_or_theme;
- devanagari;
- iast;
- plain_english_meaning;
- source_reference;
- source_level;
- usage_context;
- pronunciation_note;
- restriction_note;
- initiation_required_flag;
- public_safe_flag;
- sanskrit_review_status;
- source_review_status;
- approval_status.

No mantra may be generated by AI, recombined, beautified, paraphrased, or invented.

## 9. No Invented Mantra Doctrine

M06 explicitly prohibits:

- invented Sanskrit mantras;
- pseudo-Sanskrit phrases;
- AI-generated bija combinations;
- combining deity names with random seed syllables;
- presenting translation as mantra;
- presenting devotional English as Sanskrit mantra;
- altering a source mantra without marking it as explanation;
- claiming Vedic, Puranic, Tantric, or Siddha source without source evidence.

If a mantra is not reviewed, it must remain unavailable for selection.

## 10. Mantra Risk Classification

Mantra records must be classified before future use.

Risk classes:

- public_simple_devotional;
- public_stotra_line;
- beginner_safe_japa;
- requires_context;
- requires_pronunciation_review;
- initiation_or_diksha_required;
- tantric_or_restricted;
- not_for_public_output;
- human_review_required.

Any restricted, tantric, initiation-required, or unclear mantra must not be shown in public or automated subscriber output.

## 11. Selection Input Doctrine

Future selection logic may consider only consented and reviewed inputs.

Allowed input categories:

- generic date;
- weekday;
- tithi;
- festival or observance context;
- language preference;
- script preference;
- user spiritual preference;
- user-selected deity preference;
- user-selected no-preference status;
- profile completeness label;
- uncertainty label.

Restricted input categories requiring explicit consent:

- name;
- DOB;
- birth time;
- birth location;
- current location;
- historical interaction preference.

Prohibited input categories:

- inferred caste;
- inferred religion;
- inferred ethnicity;
- inferred health status;
- inferred political belief;
- inferred sexuality;
- inferred financial status;
- hidden behavioural targeting.

## 12. Selection Priority Doctrine

Future selection should follow safe priority:

1. safety gate;
2. consent gate;
3. source gate;
4. Sanskrit review gate for mantra;
5. restriction gate for mantra;
6. profile uncertainty gate;
7. symbolic rule-family match;
8. fallback rule;
9. explanation label;
10. audit trace.

If any required gate fails, the output must fall back to generic non-personalized guidance or no output.

## 13. Fallback Doctrine

Fallback outputs must be safe.

Allowed fallback types:

- no personalized number available;
- generic symbolic number;
- generic symbolic colour;
- simple reviewed devotional mantra;
- no mantra today;
- reflection-only guidance.

Fallback must not expose private data or imply failure.

## 14. Explanation Label Doctrine

Future outputs must explain the basis without overclaiming.

Allowed labels:

- selected from symbolic daily rotation;
- selected from reviewed festival context;
- selected from user preference;
- selected from consented profile inputs;
- selected from reviewed mantra source;
- generic non-personalized fallback.

Not allowed:

- guaranteed lucky result;
- removes all obstacles;
- must be chanted or harm will occur;
- secret remedy;
- astrologically guaranteed;
- divine certainty.

## 15. Output Contract Doctrine

Future output must separate values from explanations and review status.

A future output should preserve:

- guidance_type;
- selected_value;
- selection_basis;
- source_reference if applicable;
- consent_basis;
- personalization_level;
- confidence_label;
- explanation_label;
- safety_label;
- review_status;
- public_output_allowed.

This is not runtime code.

## 16. Audit Trace Doctrine

Every future selection must preserve an audit trace.

Minimum trace fields:

- module_id;
- selection_id;
- timestamp;
- profile_input_refs;
- consent_refs;
- rule_family;
- source_refs;
- fallback_used;
- restriction_flags;
- review_status;
- output_safety_status.

M06 does not implement this trace. It defines the requirement.

## 17. Validation and Learning Doctrine

M06 inherits M04A.

Fortnightly or monthly validation should review:

- mantra source correctness;
- Devanagari and IAST integrity;
- unsafe mantra flags;
- repeated or stale number/colour rotations;
- user feedback;
- fallback frequency;
- mismatch between rule and explanation;
- inappropriate personalization;
- privacy concerns.

Any correction must be recorded through the learning register and database tuning backlog.

## 18. Safety Doctrine

M06 prohibits deterministic, coercive, fear-based, or manipulative guidance.

Not allowed:

- This mantra guarantees wealth.
- This colour will cure illness.
- This number will prevent harm.
- Your DOB requires this remedy.
- Chanting is mandatory.
- Ignoring this will cause bad luck.
- This replaces professional advice.

Allowed:

- Symbolic daily suggestion.
- Reviewed devotional mantra.
- Optional reflective guidance.
- Non-deterministic spiritual encouragement.

## 19. Premium Gating Doctrine

M06 does not implement premium gating.

Future premium gating must not hide source status, safety warnings, or consent requirements.

Premium outputs must still follow the same Sanskrit, source, privacy, and safety rules.

## 20. Future Internal Selection Record Contract

A future selection record should include selection_id, guidance_type, selected_value, selection_basis, rule_family, input_refs, consent_refs, source_refs, sanskrit_review_status, restriction_flags, personalization_level, fallback_used, explanation_label, safety_label, review_status, and public_output_allowed.

This is not runtime code.

M06 is complete when lucky number methodology, lucky colour methodology, mantra selection methodology, no invented mantra doctrine, mantra risk classification, input doctrine, selection priority doctrine, fallback doctrine, explanation label doctrine, output contract, audit trace, validation-learning doctrine, safety doctrine, and premium gating doctrine are documented.

M06 must provide a machine-readable registry and validator confirming that no runtime, Auth, Supabase, payment, external API fetch, lucky number output, lucky colour output, mantra output, subscriber guidance, public guidance, DOB prediction, Kundli generation, or automatic remedy is introduced.
